## 1. 프로젝트의 본질

이 프로젝트는 **단순 홈페이지 개편이 아니다.** 교회의 디지털 운영 기반을 정리하고 확장 가능한 플랫폼 구조를 준비하는 작업이다.

> "지금은 홈페이지를 만들지만, 구조는 향후 플랫폼 확장까지 고려한 형태여야 한다."
> — 기쁨의교회 담당자 문서

**현재 코드베이스 평가:** 이 방향에 **부분적으로 부합**한다.
- ✅ 프론트/백엔드 분리 구조 (React + Express + tRPC)
- ✅ API 기반 구조 (tRPC)
- ✅ 데이터 구조 독립성 (Drizzle ORM)
- ✅ 향후 모바일 앱 연동 가능한 API 구조
- ⚠️ 교인 인증 구조가 아직 완성되지 않음
- ⚠️ 권한 체계가 4단계 중 2단계만 구현됨

## 2. 사용자 정의 및 권한 체계

### 담당자 문서 기준 (목표 상태)

| 사용자 구분 | 주요 권한 | 인증 방식 |
|---|---|---|
| 비회원 (Guest) | 홈페이지 기본 열람 | 없음 |
| 일반회원 (외부 포함) | 일부 신청/예약/기본 기능 | 이메일 회원가입 |
| 교인 (인증회원) | 교인 대상 기능 전체 | 휴대폰 인증 + 교인 DB 매칭 |
| 관리자/운영자 | 콘텐츠 및 시스템 관리 | 별도 관리자 로그인 |

### 현재 코드베이스 구현 상태

| 사용자 구분 | 구현 상태 | 코드 위치 |
|---|---|---|
| 비회원 | ✅ 완료 | `publicProcedure` |
| 일반회원 | ✅ 완료 | `church_members` 테이블 + `memberProtectedProcedure` |
| 교인 (인증회원) | ⚠️ **구조는 있으나 인증 미완성** | `church_members.status` (pending/approved/rejected) |
| 관리자 | ✅ 완료 | `adminProcedure` + `users.role = 'admin'` |

### 핵심 미완성 항목

**교인 인증 흐름 (담당자 문서 요구사항)**

```
기존 교인: 휴대폰 인증 → 교인 DB(요람) 자동 매칭 → 교인 권한 부여
신규 교인: 행정실 등록 → 등록 완료 후 기존 교인과 동일한 인증 절차
```

**현재 구현된 흐름**

```
회원가입: 이메일 + 비밀번호 → 관리자 수동 승인 → 교인 권한 부여
```

**차이점:** 휴대폰 인증 및 교인 DB 자동 매칭이 없다. 현재는 관리자가 수동으로 승인하는 방식이다. 이는 운영 부담이 크다.

**향후 개선 방향:**
1. 휴대폰 본인인증 API 연동 (KG이니시스, NICE 등)
2. 교인 DB(요람 엑셀/CSV)를 `church_members` 테이블에 사전 등록
3. 회원가입 시 이름+연락처로 자동 매칭 → 자동 승인

## 5. 기술 스택 및 확장성 평가

### 현재 기술 스택

| 영역 | 기술 | 담당자 요구사항 충족 여부 |
|---|---|---|
| 프론트엔드 | React 19 + TypeScript + Tailwind 4 | ✅ |
| 백엔드 | Node.js + Express + tRPC | ✅ |
| DB | MySQL (Drizzle ORM) | ✅ |
| 인증 | JWT (성도) + 환경변수 (관리자) | ⚠️ 휴대폰 인증 미구현 |
| 파일 저장 | S3 (Manus 내장) | ✅ (외부 이전 시 교체 필요) |
| 앱 연동 가능성 | tRPC API 공개 | ✅ |

### 모바일 앱 연동 가능성

현재 tRPC API 구조는 웹 전용이지만, **REST API 또는 별도 API 레이어를 추가하면 모바일 앱에서도 동일한 DB를 사용할 수 있다.** 믿음PLUS 앱과의 연동을 위한 `faithPlusUserId` 컬럼이 이미 DB에 존재한다.

## 6. 즉시 해결해야 할 문제 (우선순위 순)

### 🔴 긴급 (코드 품질 / 보안)

1. **`as any` 34개 제거** — 타입 안전성 붕괴. 런타임 오류의 원인.
2. **파일 업로드 서버 검증 없음** — 악의적 파일 업로드 가능. `server/routers/cms/upload.ts`에 파일 타입/크기 검증 추가 필요.
3. **빈 `catch {}` 블록** — 오류가 조용히 사라짐.

### 🟡 중요 (기능 완성도)

4. **교인 인증 흐름 미완성** — 현재 관리자 수동 승인 방식. 휴대폰 인증 연동 또는 최소한 자동화 개선 필요.
5. **갤러리 사진 교체 버그** — CDN URL 확인 필요.
6. **블록 에디터 저장 후 자동 갱신 안 됨** — `invalidate` 처리 누락.

### 🟢 중기 과제 (기능 추가)

7. 카카오 소셜 로그인
8. 비밀번호 찾기/재설정
9. 핵심 라우터 테스트 코드 작성 (예약, 성도 관리, 메뉴)
10. 결제 시스템 (헌금, 시설 대여) — PG 연동 필요

## 7. 인프라 방향

담당자 문서는 **하이브리드 구조(교회 서버 + 클라우드)** 를 제안한다.

현재 코드베이스는 이 방향에 준비되어 있다.

- **DB 이전:** `DATABASE_URL` 환경변수 1개 교체로 가능
- **서버 이전:** Node.js 표준 구조, 어느 서버에서든 실행 가능
- **파일 저장 이전:** `server/storage.ts`의 S3 설정을 외부 S3로 교체 필요

자세한 이전 절차는 `DEPLOYMENT.md` 참조.

담당자 문서는 **기성세대(단순/안정)** 와 **청년/다음세대(모바일/참여형)** 를 구분한다.
