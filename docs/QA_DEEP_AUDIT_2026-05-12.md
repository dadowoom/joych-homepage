# 기쁨의교회 홈페이지 심층 검수 계획서

작성일: 2026-05-12  
기준 브랜치: `origin/main`  
대상 운영 도메인: `https://dadowoomtest.co.kr`

## 1. 검수 목적

기쁨의교회 홈페이지가 납품 가능한 수준으로 동작하는지 확인한다. 이번 검수는 디자인 변경이 목적이 아니며, 기존 UI를 유지한 상태에서 사용자가 실제로 누르는 버튼, 링크, 폼, 회원 기능, 관리자 CMS 흐름이 기능적으로 안전한지 점검한다.

## 2. 검수 원칙

- 기존 화면 디자인, 색상, 레이아웃, 문구 톤은 임의로 바꾸지 않는다.
- 확실히 고장인 항목과 개선 권장 항목을 분리한다.
- 서버/DB 보안 잠금 작업은 별도 단계로 두고, 이번 문서는 홈페이지 동작 검수에 집중한다.
- 실제 운영 사이트에서 확인 가능한 항목과 코드로만 확인 가능한 항목을 구분한다.
- 버튼이 아직 실제 접수 API와 연결되지 않았다면 “무반응”이 아니라 명확한 안내가 나와야 정상으로 본다.
- 관리자/회원 인증이 필요한 기능은 비로그인 상태, 로그인 상태, 권한 부족 상태를 분리해서 본다.

## 3. 정상 판정 기준

| 영역 | 정상 기준 |
| --- | --- |
| 내부 링크 | 클릭 시 404가 아닌 의도한 화면으로 이동한다. |
| 외부 링크 | 새 창 열림 여부와 `noopener noreferrer` 처리가 적절하다. |
| 버튼 | 클릭 시 화면 이동, 저장, 안내, 모달, 토스트 중 하나가 명확히 발생한다. |
| 폼 | 필수값 누락/형식 오류/성공/실패 상태가 사용자에게 보인다. |
| 회원 기능 | 회원가입, 로그인, 로그아웃, 내 정보 접근 흐름이 끊기지 않는다. |
| 시설 예약 | 로그인 요구, 날짜/시간 선택, 중복 방지, 신청 완료/실패 안내가 동작한다. |
| 관리자 CMS | 생성/수정/삭제/정렬/업로드 후 공개 화면 반영 여부가 일관적이다. |
| 모바일 | 메뉴 열기/닫기, 하위 메뉴, 버튼 탭, 가로 넘침이 없어야 한다. |
| 오류 화면 | 없는 주소, 권한 없음, 데이터 없음, 로딩 중 상태가 어색하게 방치되지 않는다. |

1. 코드 기반 정적 분석

2. 자동 실행 검증
   - TypeScript 타입 검사
   - 테스트 실행

   - 서버 번들 build

3. 실제 사이트 검증
   - 운영 도메인 HTTP 상태 확인
   - 주요 공개 페이지 직접 요청
   - SPA 하위 주소 새로고침 확인
   - 대표 링크/버튼 동작 확인

4. 결과 정리
   - 발견사항을 P0~P3으로 분류
   - 바로 수정 가능한 항목과 별도 정보가 필요한 항목 구분

## 11. 결론

현재 상태는 “사이트가 전체적으로 뜨고, 주요 라우팅과 CMS 기반 메뉴는 깨지지 않는다”까지는 확인됐다. 즉, 접속/라우팅/빌드 수준의 큰 장애는 없다.

다만 납품 기준으로는 아직 바로 완료라고 보기 어렵다. 특히 아래 2개 축은 먼저 처리해야 한다.

1. 보안 P0: 레포에 올라간 운영 비밀값 제거 및 비밀번호/토큰 교체
2. 기능 P1: 실제 저장되지 않는 접수형 폼과 무동작 다운로드 버튼 정리

다음 수정 단계에서는 UI 디자인을 바꾸지 않고, 기존 화면의 버튼/폼 뒤 동작만 보강하는 방향으로 진행하는 것이 맞다.
