# 기쁨의교회 홈페이지 고도화 작업 계획

작성일: 2026-05-27  
대상 브랜치: `codex/social-login`  
운영/심사 도메인: `https://dadowoomtest.co.kr`

> 2026-06-03 최신 작업 기준은 `docs/NEXT_WORK_2026-06-03.md`를 우선 참조한다. 현재 이어받을 브랜치는 `codex/signup-hardening`이며, 이 문서는 2026-05-27 당시의 계획 기록이다.

## 목적

기쁨의교회 홈페이지를 납품 가능한 운영형 사이트로 고도화한다. 이번 계획의 핵심은 기존 디자인, 색상, 배치, 글자 톤을 유지하면서 뒤쪽 시스템 안정성, 관리자 운영성, 회원/권한 흐름, 콘텐츠 신뢰도, 보안과 배포 품질을 끌어올리는 것이다.

## 작업 원칙

- 기존 UI/디자인/레이아웃/색상/타이포그래피는 임의 변경하지 않는다.
- 사용자가 누르는 버튼, 폼, 메뉴, 관리자 저장 흐름이 실제 동작하거나 명확한 안내를 제공해야 한다.
- 관리자 CMS에서 만든 데이터가 공개 화면의 기준 데이터가 되도록 한다.
- 운영 서버와 DB의 민감 정보는 문서와 커밋에 남기지 않는다.
- 기능 추가 후 `pnpm check`, `pnpm test`, `pnpm build`를 기본 검증으로 삼는다.
- 배포 전후로 운영 도메인에서 핵심 기능을 직접 확인한다.

## 현재 상태 요약

| 영역 | 상태 | 메모 |
| --- | --- | --- |
| 공개 사이트 | 기본 라우팅과 주요 페이지 응답은 정상 | SPA 하위 경로 새로고침도 운영 가능 상태 |
| 성도 간편가입 | Google/Kakao 가입 및 로그인 정상 확인 | 승인 대기/승인 흐름은 계속 회귀 검증 필요 |
| 관리자 성도 관리 | 보기 분류, 탈퇴 처리, 승인 관리 흐름 보강됨 | 소셜 가입자가 관리자 목록에 반영되는지 계속 확인 |
| 카카오맵 | 운영 화면에서 실제 지도 표시 정상 확인 | SDK 도메인 설정 완료 |
| 히어로 영상 | 프론트 로딩/새로고침 로직은 보강됨 | 운영 DB에 남은 과거 영상 데이터 정리가 필요할 수 있음 |
| 푸터 사업자 정보 | 카카오 심사 대응용 운영주체 정보 표시 보강됨 | 모바일 줄바꿈/가독성 최종 확인 필요 |
| 보안 | 앱 레벨 보안 보강 다수 완료 | 운영 비밀번호/토큰 교체, SSH/DB 접근 제한은 최종 전환 때 필수 |
| CMS | 메뉴/콘텐츠/공지/시설 등 주요 관리 기능 있음 | 하드코딩 페이지와 CMS 페이지의 기준 통합이 남음 |

## 우선순위

### P0. 납품 전 반드시 막아야 할 항목

| 항목 | 목표 | 완료 기준 |
| --- | --- | --- |
| 카카오맵 실제 표시 | 오시는 길 페이지 진입 시 지도 화면이 바로 보인다 | SDK 200 응답, 지도 DOM 렌더링, 길찾기 버튼 동작 확인 |
| 히어로 영상 최신화 | 홈 첫 화면에 운영자가 의도한 최신 영상만 나온다 | 운영 DB 히어로 슬라이드 목록 정리, 새로고침/전환 시 이전 영상 재노출 없음 |
| 무동작 버튼 정리 | 클릭 가능한 버튼은 모두 동작하거나 명확히 안내한다 | PDF/CI 다운로드 버튼은 준비 중 안내 처리, 나머지 메뉴 링크 전수 점검 계속 |
| 접수형 폼 실제 처리 | 새가족/기도요청 등 제출형 화면이 허위 완료를 띄우지 않는다 | 기도 요청/새가족 등록 문의 DB 저장 및 관리자 접수 관리 연결 |
| 관리자 저장 후 공개 반영 | 메뉴/콘텐츠/시설/공지 수정 후 공개 화면이 일관되게 갱신된다 | invalidate/refresh 흐름 확인, 운영 화면 회귀 테스트 |
| 회원 승인 게이트 | 소셜 가입자는 승인 전 제한되고, 승인 후 접근 가능하다 | Google/Kakao/일반 가입 모두 같은 승인 정책 확인 |
| 백업 가능 상태 | 배포 전 소스/DB 백업이 가능하다 | 백업 스크립트 실행 확인 또는 운영 절차 문서화 |

### P1. 납품 품질을 확실히 올리는 항목

| 항목 | 목표 | 완료 기준 |
| --- | --- | --- |
| 섬기는 사람들/부교역자 CMS화 | 목회자/부교역자 정보를 관리자에서 등록하고 공개 화면에 표시한다 | 기존 좌측 메뉴/서브페이지 레이아웃 유지, 임의 등록/수정/삭제 가능 |
| 선교보고 작성 권한 | 지정된 성도만 선교보고를 작성하고 사진을 올릴 수 있다 | 관리자 권한 부여, 작성/수정/삭제, 사진 업로드, 공개 상태 관리 |
| 팝업/공지 배너 | 교회 행사/긴급공지/신청 안내를 관리자에서 노출 제어한다 | 노출 기간, PC/모바일 표시, 오늘 하루 보지 않기 |
| 새가족 안내 동선 | 처음 방문자가 예배/등록/교육/문의 흐름을 바로 이해한다 | 기존 톤 유지, 주요 CTA는 실제 동작 또는 명확한 안내 |
| 조이풀TV 분류 강화 | 예배영상이 운영형 아카이브처럼 분류된다 | 주일예배/찬양/특별집회/교회학교 등 분류와 관리자 관리 |
| 사이트 운영 정보 정리 | 심사자와 방문자가 운영주체를 바로 확인한다 | 푸터 가독성, 모바일 줄바꿈, 사업자 정보 노출 확인 |

### P2. 운영 안정성과 확장성

| 항목 | 목표 | 완료 기준 |
| --- | --- | --- |
| 사이트 검색/빠른 링크 | 방문자가 메뉴명을 몰라도 주요 정보를 찾는다 | 검색 대상/결과/빈 상태 UX 확인 |
| 주보/행사 캘린더 | 반복 방문 콘텐츠를 쉽게 찾는다 | 관리자 등록, 목록/상세, 기간별 표시 |
| 성도 마이페이지 확장 | 내 예약, 가입 상태, 권한, 신청 내역을 확인한다 | 개인정보 노출 최소화, 승인 상태 안내 |
| 감사 로그 | 관리자 주요 변경 이력을 추적한다 | 누가/언제/무엇을 바꿨는지 기록 |
| 배포 자동화 개선 | 수동 서버 작업을 줄인다 | 백업, 빌드, 재시작, 헬스체크 순서 자동화 |

## 실행 단계

### 1단계. 운영 장애 후보 즉시 정리

목표: 사용자가 지금 바로 고장으로 느낄 수 있는 부분을 먼저 막는다.

- [x] 카카오맵 렌더링 최종 확인
- [ ] 홈 히어로 영상 DB 상태 확인 및 과거 영상 정리
- [ ] 공개 페이지 버튼/링크/폼 전수 스캔
- [x] 교회백서/CI 무동작 다운로드 버튼을 명확한 안내 처리로 전환
- [x] 새가족/기도요청 폼을 실제 DB 저장 및 관리자 접수 관리로 연결

### 2단계. CMS 기준 데이터 통합

목표: 코드에 박힌 페이지와 관리자 생성 페이지가 따로 노는 문제를 줄인다.

- [ ] 메뉴의 기준 데이터를 관리자 CMS로 통일
- [ ] 하드코딩 페이지 중 운영자가 자주 바꿀 내용을 CMS 페이지로 이전
- [ ] 섬기는 사람들/부교역자 등록 구조 설계
- [ ] 공개 서브페이지에서 기존 좌측 메뉴 레이아웃 유지
- [ ] CMS 수정 후 공개 화면 반영 회귀 테스트

### 3단계. 회원/권한/커뮤니티 고도화

목표: 성도 회원 기능이 실제 교회 운영에 맞게 돌아가도록 한다.

- [ ] Google/Kakao/일반 가입 승인 정책 통합 확인
- [ ] 승인 대기/반려/탈퇴/활성 상태별 접근 권한 테스트
- [ ] 선교보고 작성 권한 부여 기능 추가
- [ ] 권한 있는 성도만 선교보고 작성/사진 업로드 가능하게 처리
- [ ] 관리자에서 작성자/공개 상태/삭제 상태 관리

### 4단계. 운영형 홈페이지 기능 강화

목표: 대형 교회 홈페이지처럼 반복 운영에 필요한 기능을 갖춘다.

- [ ] 관리자 팝업/공지 배너 기능
- [ ] 새가족 안내 흐름 정리
- [ ] 조이풀TV 분류/플레이리스트 운영성 강화
- [ ] 교회학교/부서 페이지의 예배시간/장소/대상/문의 표준화
- [ ] 사이트맵/오시는 길/새가족/행정지원 빠른 동선 점검

### 5단계. 보안/배포/운영 마감

목표: 납품 직전 운영 리스크를 줄인다.

- [ ] 운영 서버/DB/관리자/외부 API 비밀번호 최종 교체
- [ ] GitHub에 노출된 적 있는 토큰/키 폐기 및 재발급
- [ ] root 비밀번호 SSH 대신 배포 계정과 SSH 키 방식 검토
- [ ] DB 외부 접근 제한 또는 허용 IP 최소화
- [ ] 백업 스크립트와 복구 절차 확인
- [ ] `pnpm check`, `pnpm test`, `pnpm build` 최종 통과
- [ ] 운영 도메인 핵심 페이지 직접 확인

## 바로 다음 작업 순서

1. 홈 히어로 영상이 왜 과거 영상으로 보이는지 DB/관리자 기준 확인
2. 남은 공개 페이지 버튼/링크 전수 점검표 생성
3. 접수 관리 탭 운영 테스트 및 DB 마이그레이션 반영
4. CMS 기준 데이터 통합 항목 선별
5. 섬기는 사람들/부교역자 CMS 데이터 정리와 공개 화면 회귀 검증

## 의사결정 필요 항목

- 새가족/기도요청 폼은 실제 DB 저장 및 관리자 접수 관리 방식으로 결정했다.
- 히어로 영상 중 최종 노출할 영상 목록을 운영자가 확정한다.
- 선교보고 작성 권한을 개인 단위로 줄지, 부서/역할 단위로 줄지 결정한다.
- 최종 도메인 확정 시 Google/Kakao OAuth, Kakao Map, sitemap, robots, `PUBLIC_URL_BASE`를 한 번에 전환한다.

## 현재 결론

현재 사이트는 기본 접속과 주요 구조는 올라와 있지만, 고가 납품 기준으로는 "보이는 화면 뒤의 실제 동작"을 더 잠가야 한다. 우선은 카카오맵, 히어로 영상, 버튼/폼, CMS 기준 통합, 회원 권한 흐름을 순서대로 처리하는 것이 가장 효율적이다.
