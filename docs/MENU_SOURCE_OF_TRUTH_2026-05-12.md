# 메뉴 기준 정리 메모 (2026-05-12)

## 결론

홈페이지 상단 메뉴의 실제 기준은 DB의 `home.menus` 응답이다. 관리자가 `메뉴 편집`에서 수정하면 이 DB 메뉴가 공개 헤더에 반영된다.

코드 안의 메뉴 목록은 DB 메뉴가 비어 있거나 불러오지 못했을 때만 쓰는 폴백이며, 이제 `shared/siteNavigation.ts` 한 곳에서 관리한다.

## 정리한 기준

- 공개 헤더: `client/src/components/SiteHeader.tsx`
  - 우선순위 1: DB 메뉴 (`trpc.home.menus`)
  - 우선순위 2: `shared/siteNavigation.ts`의 기본 메뉴
- 관리자 메뉴 편집: `client/src/components/MenuEditPanel.tsx`
  - 저장 대상: DB 메뉴
  - `기존 페이지` 선택 목록: `shared/siteNavigation.ts`의 `INTERNAL_PAGES`
- 정적 코드 라우트: `client/src/App.tsx`
  - 실제 페이지 컴포넌트/기능 라우트다.
  - 공개 메뉴의 기준표로 쓰지 않는다.

## URL 원칙

- CMS 동적 페이지는 `/page/{slug}` 형태를 기본으로 사용한다.
- `/page/item/:id`, `/page/sub/:id`는 기존 링크 호환용 라우트로 남긴다.
- 2026-05-12 실제 서버 DB에서 메뉴 href 22건을 slug 기반 URL로 정규화했다.
- 관리자 직접 입력 예시는 숫자 기반 URL 대신 `/page/교회소개-담임목사-소개` 같은 slug 기반 URL로 안내한다.

## 왜 필요한가

기존에는 헤더 기본 메뉴, 홈 페이지 안의 미사용 메뉴 목록, 관리자 `기존 페이지` 선택 목록이 각각 따로 있었다. 이 상태에서는 메뉴명을 바꾸거나 페이지를 추가할 때 한쪽만 반영되어 관리자 화면과 실제 공개 메뉴가 달라질 수 있다.

이번 정리는 화면 디자인을 바꾸지 않고, 메뉴 데이터의 기준만 한 곳으로 모은 것이다.

## 다음 판단 지점

기능 페이지와 단순 소개 페이지는 구분해서 봐야 한다.

- 기능 페이지: 로그인, 회원가입, 관리자, 시설 예약, 선교보고, 조이풀TV, 게시판 등은 코드 라우트로 유지한다.
- 단순 소개/사역 페이지: 필요하면 CMS 콘텐츠로 계속 전환할 수 있다.
- 메뉴 노출 여부: 실제 공개 헤더는 관리자 메뉴 편집 화면에서 최종 결정한다.
